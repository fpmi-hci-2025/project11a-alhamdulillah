# Анализ интерфейса по модели KLM/TLM

## Введение

KLM (Keystroke-Level Model) и TLM (Touch-Level Model) - модели для предсказания времени выполнения задач на сенсорных устройствах. TLM является адаптацией KLM для жестовых интерфейсов.

## 1. Модель TLM (Touch-Level Model)

### 1.1. Временные константы операторов для сенсорных устройств

| Оператор | Описание | Время (сек) |
|----------|----------|-------------|
| T | Tap (одиночное касание) | 0.6 |
| L | Long press (длительное нажатие) | 1.0 |
| S | Swipe (свайп) | 0.8 |
| P | Pinch (сведение/разведение пальцев) | 1.2 |
| D | Drag (перетаскивание) | 1.5 |
| H | Homing (перемещение руки к устройству) | 0.4 |
| M | Mental preparation (ментальная подготовка) | 1.2 |
| R | System response (ответ системы) | 1.0 |
| C | Scroll (прокрутка) | 0.5 |
| Z | Pinch-to-zoom | 1.5 |

**Источники**:
- Li, Y., Gomes, A., Yeh, M. (2018) - Touch-Level Model for Mobile Gestures
- MDPI Journal - "Adapting KLM for Touch Interfaces"

### 1.2. Дополнительные параметры

| Параметр | Значение |
|----------|----------|
| Ввод одного символа на виртуальной клавиатуре | 0.3 сек |
| Переключение между полями | 0.5 сек |
| Визуальное сканирование экрана | 1.0 сек |
| Чтение короткого текста (10 слов) | 2.0 сек |
| Проверка результата действия | 0.8 сек |

## 2. Анализ форм мобильного приложения

### 2.1. Форма 1: Быстрый заказ "Шаурма за 5 рублей"

#### Описание задачи

С главного экрана: найти и заказать акционную шаурму за 5 рублей, добавить в корзину и перейти к оформлению.

#### Вариант 1: Через кнопку на главном экране

**Сценарий**:
1. Открыть приложение (уже на главном экране)
2. Найти кнопку "Шаурма за 5р"
3. Тап на кнопку
4. Тап на "Добавить в корзину"
5. Тап на иконку корзины
6. Тап на "Оформить заказ"

## 7. Оценка ROI оптимизаций

### 7.1. Экономия времени пользователей

**Допущения**:
- Средний заказ: 2 блюда из меню + оформление
- Пользователь делает 3 заказа в неделю
- 1000 активных пользователей в день

**Расчет**:

```
Текущее время на заказ:
- Выбор 2 блюд: 12.7 × 2 = 25.4 сек
- Оформление: 11.9 сек
- Итого: 37.3 сек

Оптимизированное время:
- Выбор 2 блюд (свайп): 8.7 × 2 = 17.4 сек
- Оформление (умные дефолты): 10.5 сек
- Итого: 27.9 сек

Экономия на один заказ: 37.3 - 27.9 = 9.4 сек (25%)

Экономия в день: 1000 пользователей × 9.4 сек = 9400 сек = 2.6 часов
Экономия в месяц: 2.6 × 30 = 78 часов
Экономия в год: 78 × 12 = 936 часов
```

**Вывод**: Оптимизации экономят **936 часов пользовательского времени в год**!
