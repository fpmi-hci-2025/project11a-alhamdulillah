# 1. Задачи тестирования и численные оценки качества

## Цель этапа

Определить задачи тестирования прототипов интерфейсов веб и мобильного приложения "Махачкала" и зафиксировать численные оценки качества разрабатываемого интерфейса.

## Контекст

Тестирование проводится на основе [прототипов из лабораторной работы №5](../design/lab5/) для следующих категорий пользователей:
- [Студенты](../research/USERS.md#профиль-1-студент-ключевой-пользователь) - основная целевая аудитория
- [Гастарбайтеры](../research/USERS.md#профиль-2-гастарбайтер-ключевой-пользователь) - ключевая аудитория
- [Администраторы](../research/USERS.md#профиль-5-исполнитель-заказов-сотрудник) - внутренние пользователи

## Задачи тестирования

### Задача 1: Оценка эффективности навигации

**Описание:** Оценить, насколько быстро и точно пользователи могут найти нужную информацию и функции в приложении.

**Целевая аудитория:** Студенты (первичные пользователи)

**Метрики:**
- Task Success Rate (успешность выполнения задач) ≥ 90%
- Time-on-Task (время выполнения задачи) ≤ 60 секунд для основных сценариев
- Количество ошибок при навигации ≤ 2 на задачу

**Критерий успеха:** 80% тестируемых пользователей выполняют контрольные задания за установленное время с первой попытки.

---

### Задача 2: Оценка понятности интерфейса

**Описание:** Проверить, насколько интуитивно понятны элементы интерфейса, терминология и структура приложения.

**Целевая аудитория:** Все категории пользователей

**Метрики:**
- Процент правильных ответов при обратной карточной сортировке ≥ 80%
- Совпадение ожиданий с фактическим поведением элементов ≥ 85%
- System Usability Scale (SUS) score ≥ 70 баллов

**Критерий успеха:** Пользователи без предварительного обучения понимают назначение 90% элементов интерфейса.

---

### Задача 3: Оценка производительности взаимодействия

**Описание:** Измерить временные затраты на выполнение типичных задач пользователями.

**Целевая аудитория:** Студенты, гастарбайтеры

**Метрики:**
- Время оформления заказа (GOMS/KLM анализ):
  - Веб-приложение: ≤ 45 секунд
  - Мобильное приложение: ≤ 35 секунд
- Время поиска ближайшего ресторана: ≤ 10 секунд
- Время добавления товара в корзину: ≤ 5 секунд

**Критерий успеха:** 75% пользователей выполняют контрольное задание за установленное время после прохождения вводного обучения (5 различных сценариев).

---

### Задача 4: Оценка соответствия стандартам

**Описание:** Проверить соответствие интерфейса стандартам и best practices для веб и мобильных приложений.

**Целевая аудитория:** Экспертная оценка

**Метрики:**
- Соответствие веб-чеклисту: ≥ 85% пунктов
- Соответствие мобильному чеклисту: ≥ 85% пунктов
- Отсутствие критических нарушений accessibility

**Критерий успеха:** Прототип соответствует 90% пунктам проверочных чеклистов.

---

### Задача 5: Оценка удовлетворенности пользователей

**Описание:** Измерить общую удовлетворенность пользователей интерфейсом и желание использовать приложение.

**Целевая аудитория:** Все категории пользователей

**Метрики:**
- System Usability Scale (SUS) score ≥ 70 баллов
- Net Promoter Score (NPS) ≥ 50
- Процент пользователей, готовых рекомендовать приложение: ≥ 70%

**Критерий успеха:** Средний балл SUS составляет 75+ баллов, что соответствует оценке "хорошо".

---

## Сценарии тестирования

### Сценарий 1: Быстрый заказ шаурмы за 5 рублей (приоритет: высокий)

**Описание:** Студент Артем между парами хочет быстро заказать шаурму за 5 рублей на самовывоз.

**Шаги:**
1. Открыть приложение
2. Найти акцию "Шаурма за 5 рублей" на главном экране
3. Добавить товар в корзину
4. Выбрать ближайший ресторан для самовывоза
5. Оформить заказ
6. Подтвердить заказ

**Ожидаемое время выполнения:**
- Веб: 40 секунд
- Мобильное: 30 секунд

**Критерий успеха:** 90% пользователей выполняют задачу с первой попытки без ошибок.

---

### Сценарий 2: Поиск и заказ конкретного блюда (приоритет: средний)

**Описание:** Пользователь хочет найти в меню шаурму с курицей и заказать её с доставкой.

**Шаги:**
1. Открыть приложение
2. Перейти в меню
3. Найти категорию "Шаурма"
4. Выбрать "Шаурма с курицей"
5. Добавить в корзину
6. Перейти к оформлению заказа
7. Выбрать доставку и ввести адрес
8. Подтвердить заказ и оплатить

**Ожидаемое время выполнения:**
- Веб: 90 секунд
- Мобильное: 60 секунд

**Критерий успеха:** 85% пользователей находят нужное блюдо за ≤ 30 секунд.

---

### Сценарий 3: Отслеживание статуса заказа (приоритет: средний)

**Описание:** Пользователь хочет проверить, готов ли его заказ к получению.

**Шаги:**
1. Открыть приложение
2. Перейти в профиль или раздел "Мои заказы"
3. Найти текущий заказ
4. Просмотреть статус заказа

**Ожидаемое время выполнения:**
- Веб: 15 секунд
- Мобильное: 10 секунд

**Критерий успеха:** 95% пользователей находят информацию о статусе заказа интуитивно.

---

### Сценарий 4: Управление меню (администратор) (приоритет: средний)

**Описание:** Администратор ресторана хочет добавить новое блюдо в меню и установить на него акцию.

**Шаги:**
1. Войти в админ-панель
2. Перейти в раздел "Управление меню"
3. Нажать "Добавить блюдо"
4. Заполнить информацию о блюде (название, описание, цена, фото)
5. Сохранить блюдо
6. Установить акцию на блюдо
7. Опубликовать изменения

**Ожидаемое время выполнения:**
- Веб: 120 секунд

**Критерий успеха:** 90% администраторов выполняют задачу без обращения к документации.

## Методология проведения тестирования

### Количество участников

- Для usability-тестирования: 5-8 человек на каждую категорию пользователей
- Для карточной сортировки: 10-15 человек
- Для SUS-тестирования: минимум 12 человек

### Условия тестирования

- Тестируемые **не знакомы** с разрабатываемым продуктом
- Уровень опытности соответствует [профилям пользователей](../research/USERS.md)
- Перед контрольным тестом проводится **вводное обучение** (5 различных сценариев)
- **Не прерывать** пользователей во время тестирования
- Фиксация действий: видеозапись экрана, время выполнения, количество ошибок

### Инструменты фиксации результатов

- Таймер для измерения времени выполнения задач
- [Калькулятор доверительного интервала](http://www.measuringusability.com/wald.htm)
- Форма для записи наблюдений
- SUS-анкета
- Видеозапись сессий (опционально)

## Критерии успешности всего тестирования

Интерфейс считается **успешно протестированным** и готовым к следующей стадии разработки, если:

1. ✅ Task Success Rate ≥ 85% по всем основным сценариям
2. ✅ System Usability Scale (SUS) ≥ 70 баллов
3. ✅ Соответствие чеклистам ≥ 85%
4. ✅ Время выполнения основных сценариев в пределах установленных норм
5. ✅ Процент совпадения ожиданий ≥ 80%

При невыполнении критериев требуется **итерация дизайна** и повторное тестирование.
